# Scope Control

Use this reference when deciding what belongs in the current milestone, what should be deferred, or how to respond when new ideas threaten focus.

## Core rule

Protect the smallest scope that can deliver a strong end-to-end user experience. Do not expand scope unless the added work clearly improves the main user journey or materially reduces delivery risk.

## Scope decision questions

Ask these in order:

1. Does this improve the primary user journey for the current milestone?
2. Is it required for the product to feel trustworthy, usable, or complete?
3. Does it unblock other critical work?
4. Can it be deferred without harming adoption, comprehension, or reliability?

If the answer to the first three questions is no, defer it.

## Cut-line framework

Classify proposed work as:

- Must have: required for the core flow to work and feel credible
- Should have: meaningfully improves the launch experience but can slip if needed
- Later: useful, interesting, or strategically valid, but not necessary for the current milestone

When timeline or capacity tightens, cut from `Later` first, then `Should have`. Protect `Must have` quality rather than shrinking its quality bar.

## One-in, one-out rule

When a new item enters an active milestone, identify what leaves:

- another feature
- a secondary refinement
- a later phase deliverable
- a date commitment, if the user explicitly prefers scope over schedule

Do not let scope grow invisibly.

## Anti-scope-creep language

Use phrasing like:

- "This is a good idea, but it belongs after the core flow is polished."
- "Adding this now increases surface area without improving the primary user outcome."
- "If we include this in the current milestone, we should defer X to protect quality."
- "This is valuable, but not on the critical path to a five-star first-use experience."

## Good cuts

Cut work that:

- serves edge personas before the primary user
- adds configuration before defaults are strong
- expands breadth before the core loop feels smooth
- improves internal neatness without visible user benefit
- creates UI complexity without meaningfully reducing user effort

## Bad cuts

Avoid cutting work that:

- fixes broken or confusing core flows
- improves reliability, clarity, onboarding, or error recovery
- resolves visible inconsistencies that damage trust
- addresses accessibility, mobile usability, or major performance issues
