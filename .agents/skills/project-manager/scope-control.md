# Scope Control

Use this reference when deciding what belongs in the current task, what should be deferred, or when new ideas threaten focus.

## Core Rule

Protect the smallest scope that delivers a trustworthy end-to-end user outcome. Do not expand scope unless the added work clearly improves the primary user journey or materially reduces delivery risk.

## Scope Decision Questions

Ask in order:

1. Does this improve the primary user journey for the current task?
2. Is it required for the product to feel trustworthy, usable, or correct?
3. Does it unblock other critical work?
4. Can it be deferred without harming adoption, comprehension, or reliability?

If the answer to the first three is no, defer it.

## Cut-Line Framework

Classify proposed work as:

- **Must have**: required for the core flow to work and feel correct
- **Should have**: meaningfully improves the outcome but can slip if needed
- **Later**: valid, but not necessary for the current task

When capacity tightens: cut from Later first, then Should have. Protect Must have quality — do not lower the quality bar to fit more scope.

## One-In, One-Out Rule

When a new item enters an active task, identify what leaves:

- another feature
- a secondary refinement
- a later deliverable
- or explicitly accept the scope increase with the user

Do not let scope grow invisibly.

## Anti-Scope-Creep Language

- "This is a good idea, but it belongs after the core flow is verified."
- "Adding this now increases surface area without improving the primary user outcome."
- "If we include this, we should defer X to protect quality."
- "This is valid, but not on the critical path to a correct, trusted outcome."

## Good Cuts

Cut work that:

- serves edge cases before the primary flow is correct
- adds configuration before defaults are strong
- expands breadth before the core loop is trusted
- improves internal neatness without visible user benefit
- creates new states or branches without materially improving the user outcome

## Bad Cuts

Never cut work that:

- fixes misleading or incorrect states (especially numbers, balances, pending counts)
- improves reliability, clarity, or error recovery in core flows
- resolves visible inconsistencies that damage trust
- addresses correctness or data integrity
